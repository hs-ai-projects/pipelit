# Pipelit 重构路线图

> 基于 22 个问题反馈 + 优化.md 综合评估整合
> 原则：先稳定后协议化；每 Stage 做完都有明确 milestone，可随时停下评估

---

## 总览

| Stage | 目标 | Milestone |
|---|---|---|
| Stage 0 | 准备测试集 + 决策日志骨架 | 改任何东西都有回归保护 |
| Stage 1 | 修 8 个真 bug | **自己敢用** |
| Stage 2 | 决策可追溯 + 状态机 | **跑两次结果可比对** |
| Stage 3 | 架构通用化 | **可以让别的项目试用** |
| Stage 4 | 协议化重构 | **别人能 review、能改、能扩** |
| Stage 5 | 体验细节 | **每天用都顺手** |
| Stage 6 | 分享准备 | **公开发布** |

## Stage 0 — 基础设施（不改业务代码）

**目标**：建立回归测试 + 决策日志规范，后续所有改动都基于这套。

### Task 0.1 — 建立回归测试用例集

**位置**：`tests/regression-cases/`

- `case-01-l2-clear.md` — 描述清晰、有截图、单文件改动（典型 L2）
- `case-02-l2-bug-with-log.md` — 有截图 + 报错时间，需要查日志
- `case-03-l3-vague.md` — 描述模糊（"优化下登录"）
- `case-04-l3-cross-module.md` — 跨前后端、文件多
- `case-05-release-clean.md` — 一次干净的双仓发版

每个 case 包含：输入（`task_id` 或 mock 文件） + 期望的关键判定（level、命中文件数、关键 audit 字段）

**验证标准**：能手动跑过一遍并记录"期望结果"

### Task 0.2 — 决策日志规范

**位置**：`docs/decision-log-schema.md`

- 定义 JSON schema：`task_id` / `skill` / `phase` / `decision` / `evidence` / `timestamp`
- 落地位置：`~/.claude/pipelit/decision-logs/<date>/<task_id>-<skill>.json`

**验证标准**：写一个 demo 日志能被 schema 校验

### Task 0.3 — git 协作约定

- 每个 Stage 一个分支：`refactor/stage-1-p0-fixes`
- 每个 Task 一个 commit

## Stage 1 — P0 真 Bug 修复（让自己敢用）

> 修完这 8 个，自己用应该不会再出"莫名其妙的事"。

### Task 1.1 — 修 commit body 收集（#20 + #22 问题 1）

**文件**：`skills/release/SKILL.md` Phase 1.3

- 改 `git log --oneline` → `git log --format="=== %h ===%n%B" --no-merges`
- 从 body 提取 `Feishu-Task: <url>` 直接作为该 changelog 的飞书链接
- 删掉"调用 list_tasks 做语义匹配"作为主路径，只作为 fallback

**验证**：跑 case-05，每条 changelog 的飞书链接来自 commit body 而非 fuzzy 匹配

### Task 1.2 — 修观测云时间校验（#1）

**文件**：`scripts/guance_api.py` + `skills/guance-log-analysis/SKILL.md`

- 脚本入口加 ISO 8601 校验，不合法直接返回 `GUANCE_TIME_INVALID:<reason>`
- SKILL.md 增加显式时间格式契约（年-月-日T时:分:秒+时区，UTC 或本地都要明确）
- feishu-dev 的 1.8d 必须用 `+08:00` 或 `Z` 结尾的 ISO 字符串

**验证**：跑 case-02，故意给个 `"2026/06/04 14:00"` 错误格式，应该报 `GUANCE_TIME_INVALID` 而非死循环

### Task 1.3 — 修日志匹配逻辑（#2）

**文件**：`skills/guance-log-analysis/SKILL.md` 静默模式 Step A/B

- 把"查 ERROR 日志"改为"查时间窗口内 **所有相关接口的请求/响应**"
- 默认带回 `status_code`、`request_payload`、`response_payload`（而非只 ERROR 行）
- Step A 失败后 Step B 也要返回正常请求样本（让 AI 能判断"是不是数据本身有问题"）

**验证**：case-02 跑完，输出里能看到"接口返回 200 但 data=[]"这种情况，不只 5xx

### Task 1.4 — 修接口路径定位（#3）

**文件**：`skills/feishu-dev/SKILL.md` Phase 1.7 + 1.8b

- 1.7 先**读截图里的 URL**（截图里的 network 面板 / url bar 优先级最高）
- 1.8b grep 改为"精确路径前缀匹配"，候选 > 3 个要求收敛到最长公共前缀
- AskUserQuestion 候选展示要列出"具体 URL"而非"接口模块"

**验证**：case-02 给一张截图含 `/api/ads/create-ads/report`，应该定位到 `/report` 子路径而非 `create-ads`

### Task 1.5 — 卡片无飞书任务降级（#4 最小集）

**文件**：`skills/release/SKILL.md` Phase "发送卡片"段 + `scripts/feishu_api.py`

- `send_release_card_with_mentions` 的 `sections[].entries[].task_id` 缺失时：
  - 不查关注人 → 不 @
  - 不拼飞书任务链接 → 标题不带 "🔗 飞书任务"
- 整个流程不阻断、不报错、不询问

**验证**：case-05 把所有 commit 的 `Feishu-Task:` 字段去掉，发版能完成，卡片正常发出（只是没 @ 人）

### Task 1.6 — 预览确认 Step（#22 问题 2 + 问题 4）

**文件**：`skills/release/SKILL.md` Phase 3 之间插入 3.5b

- 卡片构建完成后、`send_release_card_with_mentions` 调用前
- 打印：`sections` 完整文本 + 每条 `@ 关注人 open_id`（黑名单过滤后） + 图片来源
- AskUserQuestion 三选：`发送` / `修改内容` / `取消`
- 修改内容 → 让用户给出 diff，重新预览

**验证**：case-05 发版能看到完整预览，可以中途取消而不影响 commit/tag/push

### Task 1.7 — 分支策略一致性（#17 上半）

**文件**：`skills/feishu-dev/SKILL.md` Phase 3.1

- 把"对每个仓库独立执行"从段落改为**显式表格 + 强制 log**：

```
[3.1-frontend] 当前分支: master → 创建 feat/feishu-xxx
[3.1-backend]  当前分支: master → 创建 feat/feishu-xxx
```

- 任一仓库跳过分支创建必须输出 `[3.1-<repo>] 跳过原因: <reason>`
- 加入 audit trail：`branching_decision.json`

**验证**：case-04（跨前后端）跑完，git log 显示两个仓库都在 feat 分支提交

### Task 1.8 — BOT_AUTO_EXECUTE 判定明确化（#17 下半 + #19 部分）

**文件**：`skills/feishu-dev/SKILL.md` 顶部 + 各 Phase

- 在 SKILL.md 第一行加入：`[MODE-CHECK] BOT_AUTO_EXECUTE: <yes/no>`，强制 AI 启动时输出
- 各 Phase 跳过用户验证前必须 re-check 这个标志
- 不在 BOT_AUTO_EXECUTE 时，**Phase 3.5 不能省略**（哪怕 AI 自检通过也要等）

**验证**：case-03 不带 BOT_AUTO_EXECUTE 跑，Phase 3.5 必须出现 `请验证功能` 暂停

**Stage 1 完成 milestone**：跑 5 个 regression case 全部符合期望。可以打 v4.0.0 自用版本。

## Stage 2 — 决策可追溯 + 状态机（让结果稳定）

> 目标：跑两次能 diff 决策日志找出差异，context 压缩后能续上。

### Task 2.1 — L1/L2/L3 audit trail（#14 + #15）

**新文件**：`skills/feishu-dev/rules/classification.md`

- 决策树表（按优先级匹配，命中即返回）
- 每条规则带阈值 + 数据依据
- 每次判定必须输出 audit JSON：

```json
{
  "task_id": "...",
  "level": "L2",
  "matched_rule": "rule-3",
  "evidence": {"candidate_files": 2, "desc_length": 145, "has_screenshot": true}
}
```

- 落盘到 `~/.claude/pipelit/decision-logs/`

**修改**：`skills/feishu-dev/SKILL.md` Phase 1.3 从散文换成"按 classification.md 决策"

**验证**：case-03 跑两次，audit JSON 完全一致（除 timestamp）

### Task 2.2 — release 状态机 + resume（#19 主要）

**新文件**：`skills/release/state-machine.md`

- 显式状态：`IDLE → PREVIEWED → COMMITTED → PUSHING → PUSHED / PARTIAL_PUSHED → COMPLETED`
- 每次状态转换写 `<outputDir>/.release-state.json`
- 新增触发词："继续发版" / "/release resume"
- partial_release 可恢复

**修改**：`skills/release/SKILL.md` Phase 3 拆为对应状态机节点

**验证**：跑 case-05 到 PUSHING 中断（模拟网络故障 backend push 失败），重启会话说"继续发版"，能从 PARTIAL_PUSHED 继续

### Task 2.3 — 版本号判定精细化（#21）

**新文件**：`skills/release/rules/version-bump.md`

- 主规则表带"警告条件"
- 检测删除公共导出：`git diff -- '*.ts' | grep '^-\s*export'`
- 检测 API path 删除：`git diff -- '*.py' | grep -E '^\-\s*@(app|router)\.'`
- 命中警告 → Phase 2 预览强制显示 ⚠️ + 让用户选择保持还是升级

**修改**：`skills/release/SKILL.md` Phase 1.3 引用此文件

**验证**：构造一个删除 export 但 commit 是 `fix:` 的场景，应该建议 patch 但强制警告

### Task 2.4 — context 压缩续接机制（#19 补充）

**新文件**：`skills/feishu-dev/protocols/resume.md`

- 每个 Phase 开始 / 结束写 `<workdir>/.feishu-dev-state.json`
- 包含：`current_phase` / `task_id` / `branch` / `pending_actions`
- 启动时如果发现 state 文件且不超过 24h，AskUserQuestion 询问是否续接

**验证**：跑 case-04 跑到 Phase 3.5 后清空对话上下文，重新打开应该能恢复

**Stage 2 完成 milestone**：同一个 task 跑两次，能用 diff 工具对比 decision-log，证明决策稳定。

## Stage 3 — 架构通用化（可以让别的项目试用）

> 目标：从"otsan 的项目专用"变成"任何 FastAPI+Vue 项目都能用"。

### Task 3.1 — log-provider 抽象（#7）

**新文件**：

- `skills/log-provider/SKILL.md` — 接口定义
- `scripts/log_providers/guance.py`（现有移到这里）
- `scripts/log_providers/noop.py`（无日志源时的兜底）
- 配置文件加 `logProvider: "guance" | "sentry" | "datadog" | "noop"`

**修改**：`skills/feishu-dev/SKILL.md` 1.8d 调用 log-provider 而非直接调 guance

**验证**：把 config 改成 `logProvider: "noop"`，跑 case-02 也能完成（只是没日志辅证）

### Task 3.2 — 配置三层化（#8）

**新文件**：`docs/config-hierarchy.md`

- L1 用户级（`~/.claude/pipelit/config.json`）：凭据、log_provider 默认值
- L2 项目级（`<project>/.pipelit/config.json`）：repo paths、release config
- L3 仓库级（每个 repo 的 override，可选）：precheck 命令、特殊规则
- 合并策略：L3 > L2 > L1

**修改**：

- `scripts/feishu_api.py` 增加 `load_merged_config()`
- 所有 SKILL.md 引用 config 的地方改为"按三层加载"

**验证**：在新项目跑 release，只要项目级有 paths 就能用全局凭据

### Task 3.3 — 视频附件下载（#9）

**修改**：`scripts/feishu_api.py` 的 `get_task_full`

- 检测 attachments 里的 `file_type: "video"`
- 下载到本地 `~/.claude/pipelit/attachments/<task_id>/`
- 返回 `videos[].path` 给 SKILL.md
- 不抽帧、不 AI 分析，告知用户"已下载到 X，请人工查看"

**验证**：找一个含视频附件的 task，能下载且报告路径

### Task 3.4 — 卡片完整通用化（#4 完整版）

**修改**：`skills/release/SKILL.md` + `scripts/feishu_api.py`

- 抽出 `card-builder.py`：输入 sections，输出 lark_md
- 所有"飞书 / 任务链接 / @ 关注人 / 图片生成 / Wiki 链接"全部独立 if，缺一项不影响其他
- 增加 `cardFeatures: {"linkTask": true, "atFollower": true, "image": true}` 配置

**验证**：把所有 cardFeatures 关掉，发版能生成纯文字 lark_md 卡片

## 推荐执行顺序

```
今天:        Stage 0（半天搭好测试集）
本周:        Stage 1（最关键，做完先升一个版本自用）
下周:        Stage 2（决策稳定性）
              ↑ 这里是天然休息点，可以观察 1-2 周自用情况
之后:        Stage 3 → 4 → 5 → 6
```

**关键节制原则**：

- 每个 Stage 做完都先停下来用一周，找出新问题
- Stage 1 完成时就可以打一个 v4.0.0（major，因为决策格式变了）
- 不要一口气 Stage 1-6 全做完才发版本
