# pipelit 改造计划

> 根因：所有问题都源于 skill 是「散文」不是「契约」。  
> 原则：分批改，每批完成后有可感知的改善，bot 暂时不在范围内。

---

## 综合判断

`优化.md` 的方向是对的：不要继续堆长 prompt，而是把 pipelit 变成「状态机 + 规则表 + 协议 + 回归用例」。这不是过度工程，因为现在的问题已经不是某一个提示词没写好，而是缺少可追溯、可恢复、可测试的执行契约。

优先级需要略微调整：先修每次都会踩的正确性问题，再做规则稳定性，最后再做漂亮的工程化拆分。否则很容易把目录拆得很好看，但下次发版还是丢 commit body、卡片还是误 @、context 压缩后还是漏验证。

---

## 改造原则

- 人工确认只出现在风险边界：Plan 确认、用户功能验证、发卡片预览、破坏性/不可恢复操作。
- 其他阶段应静默工作：分析、日志查询、图片生成、@ 人查询、任务链接解析失败都不能频繁弹窗。
- 所有 AI 判断必须留下证据：L2/L3、接口候选、版本号 bump、日志窗口、卡片 @ 人都要有 audit trail。
- 飞书任务链接、观测云、图片生成、@ 人、视频附件都只能是增强能力，不能成为主流程硬依赖。
- 每个流程必须可恢复：context 压缩、网络失败、partial release、用户中断后都能从状态文件继续。

## Batch 0 — Quick Wins（今天，~2-3小时）

**目标：修掉每次用都踩的 bug，今天就见效**

- [ ] **0.1** `git log` 改用 `--format="=== %h ===%n%B"` 解析完整 commit body
  - 解决：#20（commit body 里的 Feishu-Task 链接被丢弃）、#22-问题1
- [ ] **0.2** 新增 `.pipelit-state.json`，持久化关键状态（已切环境 / 已验证 / 当前 phase）
  - 解决：#19（context 压缩后漏验证步骤）
- [ ] **0.3** hook 分两档：`confirm`（需要人工确认才弹）vs `notify`（静默记录），其余阶段不打扰
  - 解决：#5
- [ ] **0.4** feishu-dev 验证闸口硬化：非 `BOT_AUTO_EXECUTE` 时，用户功能验证未通过不得 commit/push/complete_task
  - 解决：#17（沟通过程中跳过用户测试验证）、#19（context 压缩后漏验证）
- [ ] **0.5** 前后端仓库分支策略显式化：每个 repo 独立检查/创建任务分支，禁止一个 repo 在 master 改、另一个 repo 开分支
  - 解决：#17（前端开分支，后端直接 master 改）

## Batch 1 — 稳定性（本周，~3-4小时）

**目标：同一任务跑两次结果一样**

- [ ] **1.1** L2/L3 改为决策表（关键词 + 文件数阈值），每次判定输出 audit trail JSON
  - 解决：#14（换 AI 模型判断飘）、#15（同任务两次结果不一样）
- [ ] **1.2** 飞书任务链接改为可选输入，缺失时降级展示 commit message，不阻断流程
  - 解决：#4（没有飞书链接就跑不通）
- [ ] **1.3** release 卡片发送前增加预览确认步骤（展示内容 + 飞书链接 + @人列表）
  - 解决：#22-问题2（发完才知道内容对不对）、#22-问题4（不知道 @ 了谁）
- [ ] **1.4** 增加 deterministic replay：同一任务的分类、计划、候选接口、卡片结构输出为固定 JSON，可离线比较
  - 解决：#15（同一个任务跑两次结果是否一样）
- [ ] **1.5** 速度预算：简单任务走 fast path，只读必要文件；复杂任务才扩展检索和日志分析
  - 解决：#6（简单任务也分析很久、处理很久）

## Batch 2 — 通用化（下下周，~半天）

**目标：去掉硬依赖，不用观测云也能完整跑**

- [ ] **2.1** 抽 `log-provider` 抽象层：guance 是实现之一，未配置时降级让用户手动粘贴日志
  - 解决：#7（没有观测云时整个模块废掉）
- [ ] **2.2** 接口预定位改为输出候选列表让用户确认，不再 AI 自行拍板
  - 解决：#3（排查了错误的接口路径）
- [ ] **2.3** 修 guance 时间处理 + 排查逻辑（结合请求返回一起排查，不只看 error 日志）
  - 解决：#1（时间处理 bug）、#2（排查逻辑错误）
- [ ] **2.4** 日志查询输入统一为 ISO 8601 + timezone，guance adapter 内部转换为观测云语法
  - 解决：#1（AI 推日期后不符合观测云语法）
- [ ] **2.5** 日志分析从“找报错”改为“还原当时接口返回”：请求参数、响应体/状态码、业务错误、用户理解偏差都要纳入结论
  - 解决：#2（只匹配报错日志导致误判）
- [ ] **2.6** 飞书附件下载支持通用文件类型，图片进入视觉分析，视频先下载并抽首帧/元数据，必要时提示用户补充关键片段
  - 解决：#9（用户反馈是视频形式）

## 验收标准

| 能力 | 最低可接受标准 |
|------|----------------|
| release commit 采集 | 任意 commit body 中的 `Feishu-Task:` 都能被提取；无链接时不卡片、不 @、不生成假链接 |
| 用户验证闸口 | 非自动模式下，没有“用户已验证”的状态，不能 push 和 complete_task |
| L2/L3 判定 | 每次输出 `level + matched_rule + evidence`，换模型也按规则表执行 |
| 观测云/日志 | 输入时间先规范化；无观测云时可手动粘贴日志继续；结论区分“代码 bug / 数据不准 / 参数问题 / 用户理解偏差” |
| 接口定位 | 输出候选接口列表和证据，不允许只凭 AI 语义拍板单一路径 |
| 卡片发送 | 发送前能预览 changelog、飞书链接、最终 @ 人；用户确认后才发 |
| 附件 | 图片可读；视频至少能下载并给出降级提示，不因视频崩掉 |
| 恢复 | context 压缩或 partial release 后，可从状态文件恢复当前 phase |
| 速度 | 简单 L1/L2 任务不触发全量日志/全仓扫描；优先 fast path |
