## 背景

feishu-dev 处理 bug 类任务时，仅靠任务描述和截图分析问题。引入观测云日志查询后，
可以用真实的服务端报错数据辅助定位根因，提升 L2 Plan 的准确性和 L3 分析报告的说服力。

---

## 整体数据流

```
feishu-dev Phase 1（读取任务）
  ↓
Phase 1.8（新增）：Bug 日志辅助
  │
  ├─ 1.8a：语义判断是否为 bug 任务
  │         └─ 是 → 继续
  │         └─ 否 → 跳过，不影响后续
  │
  ├─ 1.8b：代码接口预定位
  │         根据描述/截图推断功能模块
  │         grep 前端代码找候选 API 路径
  │
  ├─ 1.8c：时间推断
  │         语义提取描述/截图中的时间线索
  │         fallback：任务创建时间
  │
  └─ 1.8d：调用 guance-log-analysis（静默模式）
            ┌─ 精准查询（带接口过滤）
            │   有结果 → 使用，结束
            │   无结果 → 触发兜底
            └─ 全量兜底查询
                返回摘要注入上下文
  ↓
Phase 2（L2 Plan）或 Phase 1.3 输出（L3 报告）
嵌入观测云 Log 摘要节
```

## 详细设计

### Phase 1.8a：Bug 任务判断

**不使用关键词列表**，由 Claude 语义判断：

> 这个任务的核心是"某个功能出了问题需要排查"吗？

判断依据是任务**意图**，包括但不限于：功能异常、用户投诉、页面报错、
数据不对、接口失败、体验问题、性能劣化等。

判断为 bug 任务 → 继续 1.8b。否则跳过整个 Phase 1.8。

---

### Phase 1.8b：代码接口预定位

```
1. 根据任务描述 + 截图，语义推断涉及的功能模块
2. 在前端代码目录 grep：
   - axios / fetch / request 调用里的 url 字符串
   - api.ts / request.ts / service 目录里的接口定义
3. 得到候选接口路径列表（可能为空）
```

候选接口路径供 Phase 1.8d 使用，推断不出时直接进入全量查询。

---

### Phase 1.8c：时间推断

优先级从高到低：

| 优先级 | 来源 | 示例 |
|--------|------|------|
| 1 | 描述里的明确时间线索 | "今天下午两点"、"刚才"、"昨晚" |
| 2 | 截图里可见的时间信息 | 系统时间栏、日志时间戳、界面时间 |
| 3 | 任务创建时间（fallback） | `task.created_at` |

时间窗口大小也由 Claude 语义判断，例：
- "刚才" → 往前 30 分钟
- "今天上午" → 往前 4 小时
- 无线索 → 往前 1 小时

最终得到 `start` / `end` 两个时间点。

---

### Phase 1.8d：调用 guance-log-analysis（静默模式）

**调用方式**：invoke guance-log-analysis，prompt 中包含 `GUANCE_SILENT_MODE`，
并附带参数：

```
GUANCE_SILENT_MODE
start: <推断时间>
end:   <推断时间 + 15min>
interfaces: [<候选接口路径列表>]   # 可为空
```

**懒加载兜底策略**：

```
Step 1：带接口过滤的精准查询（interfaces 不为空时）
  → 有结果 → 使用，不再执行 Step 2
  → 无结果 or interfaces 为空 → 执行 Step 2

Step 2：全量查询（不带接口过滤）
  → 从结果中找与 bug 描述最相关的报错
```

**降级处理**：

| 情况 | 处理 |
|------|------|
| guance 未配置 | 跳过，报告末尾加 `⚠️ 观测云未配置，跳过日志分析` |
| 查询无结果 | 静默跳过，不展示摘要节 |
| 查询报错（网络等） | 静默跳过，不阻塞主流程 |

---

### guance-log-analysis 静默模式

在 `guance-log-analysis/SKILL.md` 头部新增入口：

**触发条件**：prompt 包含 `GUANCE_SILENT_MODE`

**行为变化**：
- 跳过 Phase 0 配置引导交互（未配置直接返回错误码）
- 跳过 Phase 1 时间段询问
- 直接使用传入的 `start` / `end` / `interfaces` 执行查询
- Phase 3 输出精简摘要（而非完整报告）

**精简摘要格式**：

```
━━ 🔍 观测云 Log 摘要 ━━━━━━━━━━━━━━━━━
时间段: <start> ~ <end>
错误总数: <N> 条
关联接口: <接口路径，若有>

高频报错:
  • <接口> → <错误信息>  ×N 次
  • <接口> → <错误信息>  ×N 次

可能关联: <结合 bug 描述推断的关联点>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

无数据时返回：
```
该时段未发现相关错误日志（共查询 <N> 条记录）
```
