# Bug 任务判定特征表

> feishu-dev Phase 1.8a 的结构化依据。
> 判定方式：**语义理解为主，关键词为辅**。以任务整体意图为准，不依赖字符串匹配。

---

## 判定原则

**是 bug 任务**：任务核心意图是"某个已上线功能出了问题，需要定位并修复"。

**不是 bug 任务**：新功能开发、体验优化、文案修改、性能优化（无异常表现）、重构。

---

## 特征表（触发 1.8 的信号）

| 类别 | 特征描述 | 典型表述示例 |
|------|---------|------------|
| **功能异常** | 本来能用，现在用不了 | "以前能导出，现在点没反应" |
| **数据异常** | 数据不正确、不一致、丢失 | "报表数字对不上"、"列表为空但后台有数据"、"数据不准确"、"数量显示有误"、"跟实际不符" |
| **接口失败** | 接口报错、超时、返回异常 | "控制台 500 错误"、"接口返回 null" |
| **页面报错** | 白屏、JS 报错、渲染异常 | "打开页面白屏"、"控制台 TypeError" |
| **用户投诉** | 用户反馈某功能不可用 | "用户说登录不上去" |
| **体验问题（边界）** | 操作结果与预期不符 | "点保存没反应"（但性能慢不算） |

---

## 反特征（排除信号）

| 描述 | 原因 |
|------|------|
| "新增"、"支持"、"实现" | 明确是新功能 |
| "优化"、"调整"、"升级"（无异常表现） | 优化类，非修复 |
| "重构"、"拆分"、"提取" | 架构类 |
| 描述纯技术意图，无用户感知问题 | 不触发 1.8 |

---

## 调参依据

当语义模糊时（如"优化下报表接口"但附了错误截图），**截图中有报错信息则视为 bug 任务**。

```
判断优先级：截图报错 > 任务描述语义 > 关键词
```
